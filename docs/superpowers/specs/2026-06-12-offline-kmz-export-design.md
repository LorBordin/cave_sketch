# Offline-ready KMZ export for the satellite map

**Date:** 2026-06-12
**Status:** Approved — ready for implementation planning

## Problem

The satellite-map page produces three artifacts: an interactive HTML map (folium),
a KML file, and a CaveSketch-internal JSON. The goal is to deliver an artifact that
loads a cave map into common **offline** map apps — specifically **OsmAnd** and
**Locus Map**.

Two misconceptions were cleared up during brainstorming:

1. **"Offline" is about the artifact, not the basemap.** The cave drawing is just
   vector geometry with real lat/lon. The satellite *imagery* underneath is the map
   app's responsibility (OsmAnd/Locus carry their own offline maps / cached tiles).
   We only need to deliver the cave vectors in a format those apps read natively. We
   are **not** bundling satellite imagery (avoids weight and the legal grey area of
   redistributing Google tiles).

2. **The current KML already loads offline — but freezes Locus.** The real blocker
   is a performance bug, not the format.

### Root cause of the Locus freeze

The downloaded KML (`un_altro_buco_nell_acqua.kml`, generated by the current path) is
**1.9 MB with 6,211 `<Placemark>` elements**, each holding a trivial **2-point**
`<LineString>` (one survey segment), and roughly **half are reverse duplicates**
(`3→2` and `2→3` are the same edge emitted twice). Locus instantiates one interactive
map object per placemark → ~6,211 objects → UI freeze. Google Earth is a desktop GPU
renderer that batches geometry, so it is unaffected.

Neither switching to KMZ nor switching to the existing `render_to_kml` fixes this:
KMZ unzips to the same placemarks, and `render_to_kml` also emits one placemark per
segment (plus an inline `<Style>` on each, making the file *larger*).

## Current state (relevant code)

- `cave_sketch/satellite_view/map.py`
  - `draw_map(...)` — orchestrates JSON + KML + HTML export from the survey CSV.
  - `_export_to_kml(...)` — wrapper that calls the weak exporter below.
  - `create_html_map(...)` — combines **all** loaded JSON maps into the folium HTML.
- `cave_sketch/geo/kml.py::export_kml` — **weak, line-only** KML exporter (the source
  of the 6,211 placemarks). Used by: the satellite page, `tests/test_kml.py`,
  `utility_scripts/test_kml_export.py`.
- `cave_sketch/backend_renders/google_earth.py::render_to_kml` — full-fidelity
  (lines/polygons/points) but one placemark per segment. Used only by the standalone
  `create_kml.py`.
- `cave_sketch/features/render_features.py::extract_features_from_json` — backend-agnostic
  feature model (lines/polygons/points + styles) shared by the folium and KML renderers.
- `cave_sketch/style.py::STYLE_MAP` — per-type colors/weights/dashes/markers.
- No KMZ / `MultiGeometry` / `zipfile` usage exists yet.

## Goal

Deliver a single **`.kmz`** artifact from the satellite-map page that:

- carries **full fidelity** — styled survey lines, water polygons, and point features
  (blocks, ice, snow);
- **opens fast in Locus and OsmAnd** (no freeze), by collapsing the per-segment
  placemark explosion into a handful of aggregated placemarks;
- requires **no network connection** to generate or to view (the basemap is supplied
  by the target app).

## Design

### Components

Three well-bounded units:

1. **`chain_segments_by_type(lines)`** — a *pure* function (no I/O), in a new
   `cave_sketch/features/chaining.py` (sibling to `geometry.py`).
   - **Input:** the JSON map's `lines` list — entries shaped
     `{"from": {"id","lat","lon"}, "to": {"id","lat","lon"}, "type": str}`.
   - **Output:** `{feature_type: [polyline, ...]}` where each polyline is an ordered
     list of `[lat, lon]` pairs.
   - This is where the segment explosion is solved, and the one algorithmically
     interesting unit — built test-first.

2. **Upgraded `render_to_kml`** (`backend_renders/google_earth.py`) — consumes chained
   polylines + water polygons + points, emits compact KML:
   - **one shared `<Style id="...">` per feature type**, referenced via `<styleUrl>`
     (not an inline `<Style>` on every placemark);
   - **one `<Placemark>` containing a `<MultiGeometry>` per line type**, holding all
     chained polylines of that type;
   - water polygons and point features stay as **individual placemarks** (there are
     few of them — no explosion risk).

3. **`render_to_kmz(...)`** — thin wrapper that zips the KML string into a `.kmz`
   (a zip archive containing `doc.kml`) using stdlib `zipfile`.

### Chaining algorithm

Per feature type, treat the survey as an undirected graph:

1. Dedupe edges by `frozenset({from_id, to_id})` → removes reverse duplicates.
2. Build adjacency and compute node degrees. **Endpoints/junctions = degree ≠ 2.**
3. Walk maximal chains: starting from each junction/endpoint, follow through
   degree-2 nodes until reaching the next junction/endpoint; emit one polyline; mark
   edges visited.
4. Any leftover edges (pure cycles, all degree 2): pick an unvisited edge and walk
   until returning to the start; emit as a closed polyline.

Node coordinates come from the line endpoints' `lat`/`lon`. Result: thousands of
2-point segments collapse into a handful of long polylines split naturally at passage
junctions — **~6,211 placemarks → ~5–7 MultiGeometry placemarks.**

### Output structure

```xml
<Document>
  <Style id="L_wall">…red, width 2…</Style>        <!-- one shared style per type -->
  <Style id="station">…</Style>
  …
  <Placemark>
    <name>L_wall</name>
    <styleUrl>#L_wall</styleUrl>
    <MultiGeometry>
      <LineString><coordinates>…</coordinates></LineString>
      …many chained polylines…
    </MultiGeometry>
  </Placemark>
  …one per line type…
  <Placemark>…water Polygon…</Placemark>            <!-- few, kept individual -->
  <Placemark>…BLOCK Point…</Placemark>              <!-- few, kept individual -->
</Document>
```

Then zipped into `.kmz`.

### Wiring & UI changes

- `draw_map` (`satellite_view/map.py`) calls the upgraded renderer + `render_to_kmz`
  instead of `_export_to_kml`, combining **all** loaded JSON maps (matching the
  HTML map's behavior).
- Satellite page (`app/pages/2_satellite_map.py`): the third download button becomes
  **"📥 Download KMZ Map"**, producing `{sanitized_name}.kmz`.
- **Remove** the now-dead weak KML path:
  - delete `_export_to_kml` from `satellite_view/map.py` and its `geo/kml.py` import;
  - delete `cave_sketch/geo/kml.py`, `tests/test_kml.py`, and
    `utility_scripts/test_kml_export.py`.
- `create_kml.py` automatically benefits from the upgraded `render_to_kml`.

### What stays the same

- HTML (folium) and JSON exports are unchanged.
- The georeferencing / coordinate-transform pipeline is unchanged.
- `extract_features_from_json` and `STYLE_MAP` remain the shared source of styling.

## Testing & verification

- **Unit (TDD) — `chain_segments_by_type`:**
  - straight line of N nodes → 1 polyline;
  - Y-junction → 3 polylines meeting at the junction;
  - closed loop → 1 closed polyline;
  - reverse-duplicate edges (`3→2` and `2→3`) → counted once;
  - mixed types → grouped independently per type.
- **Output checks:** KML is well-formed XML; total placemark count ≪ segment count;
  one shared `<Style>` per type present and referenced.
- **Field verification (the real proof):** load the generated `.kmz` in **both Locus
  and OsmAnd** and confirm it (a) opens quickly with no freeze, and (b) renders walls,
  water, and point features correctly.

## Risks & open items

- **OsmAnd polygon fidelity:** OsmAnd historically treats KML polygons weakly (it
  leans on GPX internally), so water areas may render as outlines rather than filled
  there. Accepted baseline: lines + points render reliably in OsmAnd; **Locus gets
  full fidelity** including filled water. Confirm during field verification.
- **Streamlit offline load:** core Streamlit bundles its frontend and should run with
  no connection, but this was not empirically confirmed. To verify when running
  locally offline. (Not blocking — the KMZ generation itself makes zero network calls.)

## Out of scope

- Bundling satellite imagery / offline raster tiles (.mbtiles/.sqlitedb).
- GPX export.
- Douglas–Peucker vertex thinning (segments are already 2 points; object count, not
  vertex count, was the problem).
